Statusline Usability Review

tonone two-line CLI status bar • Nielsen heuristic evaluation • 2026-04-12

Verdict: The creator can't parse it. That's the diagnosis.

The statusline packs 14–15 data points across 2 lines using abstract symbols with no labels. It violates Recognition over Recall (Nielsen #6) at a fundamental level. The fix: group by mental model, add micro-labels, cut to what matters now.

Findings

■ Critical — Nielsen #6
Symbol soup: 6 abstract glyphs with zero labels
◆ = branch, ▲ = dirty, ⊕ = subagents, ◎ = rate limit, ↑ = ahead, ↻ = reset. These require memorizing a legend. The user — who designed this — can't parse it at a glance. That's the clearest possible signal. Symbols work only when universally understood ($ for money, % for percentage). Custom symbols create a private language.

Fix: Replace symbols with micro-labels. ▲2 → 2 dirty. ⊕1 → +1 sub. The 3–4 extra characters per segment buy instant comprehension.
▲ Major — Nielsen #8
14 segments across 2 lines overloads working memory
Miller's Law: humans chunk 7±2 items. The statusline presents 14–15 discrete data points with flat hierarchy — identical pipe separators, similar visual weight. The eye has nowhere to anchor.

Fix: Reduce to max 5 segments per line. Cut tokens (cost tells the same story), cost/line (vanity metric — show at session end), and the duplicate rate limit.
▲ Major — Nielsen #4
No semantic grouping — 4 domains share one line with identical separators
Line 1 mixes git state (branch, dirty, ahead), agent state (name, subagents), consumption (tokens, cost), and system health (context, rate limit). All separated by the same dim pipe. The brain can't chunk what looks visually identical.

Fix: Group by mental model. Line 1 = work context (what you're doing). Line 2 = resources (what's being consumed). Use visual separator hierarchy: spaces within groups, pipes between groups.
▲ Major — Nielsen #4
Rate limit appears on both lines with different formats
Line 1: ◎65% ↻2h (five-hour only, shown >50%). Line 2: 5h: 24% and 7d: 41% (both windows, always shown). Two displays for overlapping data, different thresholds for visibility, different symbols. Which one do you look at?

Fix: One location. Line 2 only. Both windows. Drop the ◎ symbol from Line 1 entirely.
▲ Major
Progressive disclosure creates layout instability
When segments hide (tokens at 0, cost at $0, agent when none), everything downstream shifts left. You can't build positional memory — "cost is always in position 6" is only true sometimes. A status bar's strength is glanceability, which requires spatial stability.

Fix: Use fixed-order groups (not segments). Groups always appear in the same position. Items within a group can still hide. Since there are only 2–3 groups per line, positional shift is minimal and predictable.
● Minor
Yellow means 3 unrelated things
Yellow = dirty files = cost >$1 = rate limit 70–90%. When the same color signals different alerts, color loses its power as a channel. Keep yellow for warnings only (cost, rate limit). Dirty count doesn't need a warning color — it's normal working state.
● Minor
"tk" is a non-standard abbreviation
In publishing, "tk" means "to come" (placeholder text). For tokens, use "tok" or just show the number without a unit label — the position makes it clear.
● Minor
Line 2 lacks a coherent organizing principle
Model name, session duration, lines changed, cost/line, rate limits. The code comment says "model + usage windows" but it's really 5 unrelated things. Cost/line is a "wow metric" (per the code comment) — interesting at session end, noise on a persistent display.

Proposed Redesign

Design Principles

1
Max 5 segments per line. If you can't fit it in 5, you're showing too much for a status bar.
2
Labels for anything non-universal. $ and % don't need labels. ▲ does. Spend 3–5 characters on a word to save the user 3–5 seconds of confusion.
3
Group by mental model, not data source. Line 1: work context (what you're doing). Line 2: resource gauges (what's being consumed).
4
Stable group positions. Groups always appear in order. Items within a group can hide. Positional shift stays within one group.
5
One source of truth per metric. Rate limit in one place. Cost in one place. No duplicates.

Line 1 — Work Context

What you're working on. Answers: which branch? any uncommitted work? who's running? what changed?

main ↑3 │ 2 dirty │ spine +1 sub │ +156 -23 lines

4 groups: git identity, working tree state, agent activity, work output. Each instantly scannable.

Line 2 — Resource Gauges

What's being consumed. Answers: which model? how long? how much spent? how much runway left?

Opus 4.6 │ 23m │ $0.38 │ ████░░░░░░ 48% │ 5h: 24% · 7d: 41%

5 groups: model, duration, cost, context window, rate limits (5h · 7d as one group). All "gauges" — things that fill up or drain.

What Changed

Removed

  • ◆ symbol before branch — position is the label
  • Token count — cost tells the same story more usefully
  • Cost/line — vanity metric, show at session end instead
  • Line 1 rate limit (◎) — duplicate of Line 2
  • ▲ ⊕ ◎ symbols — replaced with words

Added / Changed

  • "dirty" label after count
  • "+1 sub" label instead of ⊕1
  • "lines" label after +N -N
  • Lines changed moved to Line 1 (it's work context)
  • 5h/7d joined with · as one group (both are rate limits)

What's Working Well (don't change)

  • Color-escalation on context bar and rate limit — green/yellow/red/blink progression is immediately understood
  • The 10-block progress bar — universally scannable, great spatial encoding
  • Progressive disclosure concept — hiding zero-value segments is correct; the execution just needs group-level stability
  • Two-line layout — the right amount of space for a status bar
  • +N / -N for lines — matches git convention, no learning curve
  • Cost color thresholds ($1 yellow, $5 red) — sensible escalation

Top Priority Fix

If only one thing changes: replace symbols with micro-labels. This single change addresses the core complaint ("hard to understand what is what") with minimal code change. ▲2 → 2 dirty, ⊕1 → +1 sub, drop ◆. Three lines of code, immediate comprehension gain.
