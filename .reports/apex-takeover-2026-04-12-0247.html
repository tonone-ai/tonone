Key Design Decisions

  • Agents are prompts, not code. The product value is in the structured markdown that defines each agent's persona, decision framework, and workflow.
  • Self-contained agents. Each agent has its own plugin manifest, scripts, tests, and hooks. No cross-agent code dependencies.
  • Output kit as a shared contract. All 23 agents and 138 skills reference a single formatting standard (40-line CLI max, severity indicators, box-drawing).
  • Dual-location agent defs. Canonical copy in team/<agent>/agents/, working copy in root agents/. CI enforces sync.
  • Model tiering. Apex runs on Opus; specialists run on Sonnet. Cost control by design.

Findings

▲ WARNING — F1

No prompt injection defenses

No agent carries a standing instruction to treat user-provided content as untrusted. Skills like cortex-integrate and warden-audit read untrusted codebases directly into LLM context. cortex-prompt uses XML delimiters for user content but no explicit "do not override system instructions" guard.

Fix: Add a standard prompt injection defense paragraph to the agent template and backfill into all 23 agents. Add to the skill template for skills that accept external input.

▲ WARNING — F2

12 skills missing output-kit contract line

All v0.6.6 skills (touch-ui, surge-landing, prism-stack, prism-chart, pitch-landing, lens-chart, form-style, form-palette, draft-patterns, draft-landing + 2 others) lack the mandatory output-kit reference. These have a lighter body format (reference-style vs. step-by-step workflow). No CI test catches this.

Fix: Add test_skill_output_kit_contract to test_structure.py. Backfill the contract line into the 12 missing skills.

▲ WARNING — F3

No eval infrastructure for agent behaviors

cortex-eval exists for evaluating user ML features, but nothing measures whether the 23 agents themselves comply with output format, schema adherence, or decision quality. Prompt drift is invisible.

Fix: Build an agent eval suite — test output format compliance, routing accuracy, handoff schema adherence. Start with the output-kit 40-line rule and severity indicator usage.

▲ WARNING — F4

Git tags stale — release process broken

Only one git tag exists (v0.1.0) but CHANGELOG shows current version 0.6.6. No automated version bumping, no tag-on-merge. Version lives in marketplace.json only.

Fix: Add a CI job or post-merge hook that creates a git tag matching the version in marketplace.json. Backfill tags for 0.6.0–0.6.6.

▲ WARNING — F5

Sparse skill routing in CLAUDE.md

CLAUDE.md covers ~10 trigger patterns for 138 skills. High miss rate likely for mid-specificity requests. Users asking for things outside the top-10 patterns won't get routed to the right skill.

Fix: Expand routing rules or add a skill discovery mechanism. Consider a "routing index" skill that maps intents to skills programmatically.

● INFO — F6

No cost ceiling or timeout protocol for specialist dispatch

Apex dispatches specialists via the Agent tool but there's no kill switch if a specialist runs deep on what was scoped as a Small task. The S/M/L format with estimates is a good start but lacks enforcement.

Fix: Add explicit token budget guidance per S/M/L tier in the Apex prompt. Consider a meta-skill that monitors dispatch cost.

● INFO — F7

Agent tracker uses description-string matching

tonone-agent-tracker.js matches agent completion by desc string equality. If two subagents share a description prefix, the wrong one could be marked finished.

Fix: Use the agent ID (already available in hook context) for matching instead of the description string.

● INFO — F8

No SKILL.md template in new-agent scaffold

templates/new-agent/skills/SKILL_NAME/ directory exists but contains no populated SKILL.md scaffold. New agent authors must copy from an existing skill and strip it.

Fix: Add a scaffold SKILL.md with placeholder frontmatter and the output-kit contract line.

● INFO — F9

CI dependencies unpinned

pip install pytest without version pin in CI. GitHub Actions at major version (@v4) not SHA-pinned. Low risk at current scale but blocks reproducible builds.

Fix: Pin pytest version in CI. SHA-pin Actions when hardening is prioritized.

● INFO — F10

No root-level task runner

No make, just, or npm run commands at root for common dev tasks (run all tests, lint all, validate structure). Developers must know to run pytest against specific test files.

Fix: Add a Makefile or justfile with test, lint, validate targets.

Don't Touch List

Load-bearing walls of the system. Do not change without strong justification and broad review.

1. Agent definition structure

The 7-section template (Communication → Operating Principle → Scope → Platform Fluency → Workflow → Key Rules → Anti-Patterns) is consistent across all 23 agents. Changing this structure means touching every agent. Leave it alone.

2. Helm ↔ Apex handoff protocol

The 6-field brief schema with mandatory out_of_scope is the best-designed interface in the system. It works. Don't simplify or "improve" it.

3. Output kit (docs/output-kit.md)

The 40-line CLI max, severity indicators (▪ ▲ ●), and atlas-report escalation pattern are load-bearing constraints referenced by 126+ skills. Changes cascade everywhere.

4. Plugin manifest format (.claude-plugin/plugin.json)

This is the Claude Code contract. The schema is external — it's defined by the Claude Code platform, not by this project. Respect it as-is.

5. CI structural tests (test_structure.py)

These tests enforce the architectural invariants that keep 23 agents and 138 skills consistent. Add to them, never weaken them.

6. Statusline hooks

Both tonone-statusline.js and tonone-agent-tracker.js are well-written with proper null-safety, timeouts, and atomic writes. They work in production. Touch only to fix the desc-matching issue in the tracker.

7. Model tiering (Apex=Opus, specialists=Sonnet)

This is an intentional cost-control design. Don't promote all agents to Opus "for quality" — it would blow up token spend with marginal quality gain for specialist work.

Atlas — Architecture Details

Full findings

System type: Pure text/markdown-driven prompt-engineering framework. No compiled languages, no bundlers, no runtime framework.

Languages: Python 3.10+ (per-agent analyzer stubs), JavaScript/Node.js (hooks only — statusline, agent tracker). Package manager: uv (preferred) or pip.

Three-layer architecture:

  1. Root plugin (/.claude-plugin/plugin.json) — installs the full 23-agent team. Two hooks: SessionStart runs statusline installer, PostToolUse[Agent] runs agent tracker.
  2. Agent layer (team/<agent>/) — each agent is self-contained with plugin manifest, agent definition, skills, scripts, hooks, and tests.
  3. Standalone skills (skills/*/SKILL.md) — 138 skills as independent installable plugins.

Bundles: bundle/ contains thin manifests grouping agents into engineering-team (15), product-team (8), and full-team (23).

Internal coupling: Agents reference each other by name in prompts (conversational coupling only). No code-level imports between agents.

External dependencies: Effectively none. Python deps list is empty in every pyproject.toml. Node.js hooks use only stdlib.

Documentation: Good coverage — output-kit.md, naming-guide.md, agent-guide.md, skill-guide.md, architecture.md, CLAUDE.md, CONTRIBUTING.md, ROADMAP.md, CHANGELOG.md. No ADRs (appropriate for this stage).

Relay — CI/CD Details

Full findings

CI pipeline: GitHub Actions, single workflow .github/workflows/ci.yml. Runs on push to main and all PRs.

Jobs: (1) validate-structure — pytest against tests/test_structure.py; (2) lint-shell — shellcheck at WARNING severity on all setup.sh files. Agent matrix job exists but is disabled (stub tests only).

Release process: Manual. CHANGELOG.md maintained by hand. Version in marketplace.json. One stale git tag (v0.1.0) vs. actual version 0.6.6.

Hooks: No pre-commit hooks active. Only default git sample hooks.

Deployment: None. This is a plugin repo — the git repo is the artifact.

Gaps: Stale git tags, no dep caching, no branch protection documented, disabled agent matrix job is dead code.

Warden — Security Details

Full findings

Secrets: Clean. No hardcoded secrets, API keys, tokens, or connection strings. No .env files committed. .gitignore correctly excludes sensitive patterns.

Dependencies: All 23 agent pyproject.toml files declare dependencies = []. Only CI installs pytest unpinned. Low risk.

Access control: N/A — no endpoints, no HTTP servers, no network surface.

Supply chain: setup.sh scripts use local editable installs only. No curl | bash patterns. CI uses Actions at major version, not SHA-pinned. Non-urgent.

Assessment: Attack surface is effectively zero. The only real exposure is the CI pipeline itself (unpinned deps + non-SHA-pinned Actions). Both are low-severity at this scale.

Test coverage: Covers plugin.json fields, agent def existence, setup.sh existence, bidirectional sync. Does NOT cover: SKILL.md frontmatter fields, output-kit contract presence, v0.6.6 format consistency.

Hooks quality: Both hooks are well-written. Statusline: clean segment composition, null-safe, 3s stdin timeout, ANSI color helpers. Agent tracker: path traversal guard on session ID, atomic write via temp+rename, 10-min pruning, silent fail on exit. One issue: desc-string matching for completion.

Cortex — AI/Prompt Architecture Details

Full findings

Prompt architecture: Strong. All agents follow role-before-task, constraints-before-instructions, negative-examples pattern. Form agent stands out with "AI slop detection" gate and logo evaluation checklist as concrete behavioral anchors.

Orchestration: Apex dispatches via Agent tool, parallel for independent work, sequential for dependent. S/M/L format with token/cost estimates is a good cost gate. "One lateral check-in max, then escalate" prevents sprawl. Model tiering (Apex=Opus, specialists=Sonnet) is correct.

Helm↔Apex handoff: Best-designed interface in the system. 6-field schema is explicit and falsifiable. Mandatory out_of_scope with forcing function ("if you wrote none, you haven't thought hard enough"). helm-handoff validates before dispatch. One risk: open_questions can contain non-blocking feasibility asks.

Prompt risks: No injection mitigations. Skills that read untrusted codebases pipe content directly into LLM context. XML delimiters used for user content in cortex-prompt but no explicit "treat as untrusted" instruction. The review skill checks for "LLM trust boundary violations" but this is a reviewer check, not a runtime control.

Output control: Tight. Output kit is well-enforced at the prompt level. 40-line rule with atlas-report escalation is clean progressive disclosure. Communication protocol kill-list is specific enough to be enforceable. Risk: 40-line rule has no enforcement mechanism beyond prompt instruction.

Missing: No agent eval infrastructure, no cost ceiling protocol, sparse skill routing (10 patterns for 138 skills), no version field in agent definitions.
